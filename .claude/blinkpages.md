# BlinkPages platform guide — this is a BlinkPages tenant site

This repo **is a live website**, hosted and edited on the **BlinkPages** platform. Its owner runs it by asking
Claude — in the in-page **"Edit with AI"** editor or with the `/blinkpages-site-edit` skill. **Any** change,
however you arrive at it (the editor, the skill, or a direct request here), follows the same conventions. The
full editing craft lives in
[`.claude/skills/blinkpages-site-edit/docs/EDIT-METHODOLOGY.md`](skills/blinkpages-site-edit/docs/EDIT-METHODOLOGY.md).

**Who you're helping.** The owner may not be technical — they think in terms of their *site* and their
*business*, not code or git. Explain what you changed in plain language, keep their brand and voice consistent,
and make the experience smooth: no jargon, no dead ends, and always a way to preview before publishing.

**Read `reference/` first.** Before any edit, read everything in the [`reference/`](../reference/) folder — the
owner's own notes on their voice and tone, their business, their customers, and their marketing. It is **not**
published to the site; it exists to make your edits sound like *them*. Use it to match wording, tone, claims,
and audience. (A file that's still just the empty skeleton? Skip it.)

**The owner's own instructions live in `CLAUDE.md`** at the repo root (it imports this file). Follow both;
where they genuinely conflict on platform mechanics (drafts, previews, publishing, what may be edited), this
file wins — everything about voice, content, and the site itself is the owner's call.

The essentials:

- **Draft by default.** A *draft* is our word for a `draft-<slug>/v1` branch + an ordinary **open** PR off the
  live branch — a friendly name, **never** a literal GitHub draft-status PR, which can't be merged (no `--draft`,
  ever). A *variation/option* is a sibling draft in the same `draft-<slug>/…` group (its own PR **off `main`**),
  never a sub-PR off another draft. If the owner names an existing draft or option, work on that branch;
  everything else is about the live site and is delivered as a **new draft** — even small fixes. Commit straight
  to the live branch only after the owner explicitly confirms (recommend the draft). The open PR is what creates
  the preview and the editor's drafts-list entry — a branch alone is invisible. Recipe: the methodology's
  **Draft by default** + **The draft loop** sections.
- **Always end with the preview link.** A draft isn't delivered until the owner has its preview URL — the link
  in the PR's sticky "🔍 Open Preview" comment, or `https://<alias>--<tenantId>.blinkpages.dev` where `<alias>`
  is the branch with every non-alphanumeric run collapsed to `-` (so `draft-holiday/v2` → `draft-holiday-v2`;
  `tenantId`: `.migrate-to-astro/tenant.*.json`).
- **Default to the page in question.** When a request comes from — or refers to — a specific page, edit that
  page (and the components/props/data it renders) only. Go site-wide or cross-page **only** when the request
  clearly asks for it ("on every page…", "across the site…", "the footer everywhere…").
- **Variations.** **"N variations" means N options total** — build them as **flat siblings** in one
  `draft-<slug>/…` group: `draft-<slug>/v1`, `/v2`, … each a PR **`--base main`** (never a sub-PR off another
  draft), so "three variations of the home page" → **three options** `…/v1 …/v2 …/v3`. The shared `/vN` folder is
  what groups them; publishing any one goes live and discards the rest. **Never** commit competing options onto
  the live branch; give each a short, human-readable PR title — it labels that option's column in compare.
  (Full recipe: the methodology's **Multiple options** section.)
- **Respect `writableRoots`.** Only write inside the roots the tenant config declares (usually `src/` and
  `public/`). One logical concern per commit; messages like `blinkpages-ai: <what changed>`.
- **Images:** edit / replace / crop / restyle existing imagery only — never fabricate a new photo. If a new
  image is genuinely needed, stop and ask for an upload.
